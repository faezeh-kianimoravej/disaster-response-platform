# User Stories

Below is a table format for documenting all project user stories.

| ID   | Title                                | As a...   | I want to...                                                                                      | So that...                                         | Acceptance Criteria                                                                 | Priority |
| ---- | ------------------------------------ | --------- | ------------------------------------------------------------------------------------------------- | -------------------------------------------------- | ----------------------------------------------------------------------------------- | -------- |
| US1  | Incident Prioritization              | Authority | Assign priority levels to incidents                                                               | The most urgent incidents are handled first         | Incidents have priority levels; system displays incidents sorted by priority        |          |
| US2  | Incident Information Validation      | Authority | Confirm the accuracy of information for new incidents                                             | Only reliable incidents are processed               | Authority can validate or flag incidents; only validated incidents proceed          |          |
| US3  | Registered Incident Validation       | Authority | Review and verify details of ongoing incidents                                                    | Incident data remains accurate and trustworthy      | Authority can update status after validation; invalid data is flagged               |          |
| US4  | Receive Incidents from Citizens      | System    | Accept incident reports from citizens via 112                                                     | Incidents are registered and tracked efficiently    | System logs incident data from 112; incidents are visible in the system             |          |
| US5  | Receive Incidents from Professionals | System    | Accept incident reports from professionals (e.g. rangers)                                         | Incidents are registered and tracked efficiently    | System logs incident data from professionals; incidents are visible in the system   |          |
| US6  | Receive Predicted Incidents          | System    | Accept predicted incident data from research institutes or warning stations                        | Predicted incidents are registered and tracked      | System logs predicted incident data; incidents are visible in the system            |          |
| US7  | Notify Authority of New Incidents    | System    | Alert the Authority about new incidents, including urgency details                                | Authority can respond quickly to new incidents      | Authority receives notifications with urgency information for new incidents         |          |
| US8  | Disaster Notifications to Citizens   | Citizen   | Get SMS or mobile alerts about active disasters                                                   | Citizens can avoid danger and stay informed         | Citizens receive timely disaster notifications via SMS or mobile                    |          |
| US9  | Evacuation Route Notifications       | Citizen   | Get SMS or mobile alerts about evacuation routes                                                  | Citizens know where to go and stay safe             | Citizens receive clear evacuation route notifications                               |          |
| US10 | Public Announcements                 | Authority | Send important updates and announcements to the public                                            | Citizens are kept informed about critical events    | Authority can create and send announcements; citizens receive them                  |          |
| US11 | Geo-location Alerts                  | Authority | Create alerts targeted to specific geographic areas                                               | Warnings reach people in affected locations         | Authority can set geo-targeted alerts; recipients in those areas are notified       |          |
| US12 | Resource Reporting                   | Responder | Report available resources for incident response                                                  | Resource allocation can be managed                  | Responders can submit resource availability; system tracks reported resources       |          |
| US13 | Resource Assignment                  | Authority | Assign resources to specific incidents                                                            | Incidents are managed with appropriate resources    | Authority can allocate resources; assignments are tracked in the system             |          |
| US14 | Task Progress Tracking               | System    | Monitor progress of tasks related to incident response                                            | Authorities can oversee and manage incident tasks   | System tracks and displays task progress for incidents                              |          |
| US17 | Incident Chatroom Creation           | System    | Create a shared chatroom for each incident                                                        | Responders can collaborate effectively during incident response | System creates chatrooms linked to incidents; only responders assigned by authority can join and communicate |          |
| US18 | Shelter Location Display             | Citizen   | View shelter locations during incidents                                                           | Citizens can find safe places quickly               | Citizens can access shelter location information during incidents                   |          |
| US19 | Real-time Team Location Tracking     | Authority, Responder | Send and monitor live GPS location updates for response teams during incident response            | Authorities can track responder locations and improve safety and coordination        | Responders transmit GPS data; Authority can view team locations in real time; system displays real-time locations |          |
| US20 | Crisis Information Updates           | Responder | Send updates about the crisis situation as it evolves                                             | Authorities stay informed and can coordinate response | Responders can submit crisis updates; system distributes information to Authority   |          |
| US21 | Crisis Information Management        | Authority | Update and manage information about ongoing crises                                                | Information remains current and accurate            | Authority can edit crisis details; updates are reflected in the system              |          |
| US22 | Active Incident Map                  | System    | Display a map of all active incidents                                                             | Users can visualize incident locations and status   | System shows a real-time map of active incidents                                   |          |
| US23 | Receive Instructions                 | Responder | Receive instructions for incident response                                                        | Responders know what actions to take during an incident      | Responders receive clear instructions from Authority via the system                 |          |
| US24 | Accept or Discuss Task Assignment    | Responder | Confirm assigned tasks for incident response, or contact Authority to discuss | Authority knows which responder is handling each task, and special cases are managed | Responders can confirm tasks or request reassignment/denial; system updates task status and logs |          |
| US25 | Create Instructions for Responder    | Authority | Create and assign instructions to responders                                                      | Responders are guided and coordinated during incidents | Authority can create instructions; responders receive and acknowledge instructions  |          |

## Suggested additions

| ID   | Title                                | As a...   | I want to...                                                                                      | So that...                                         | Acceptance Criteria                                                                 | Priority |
| ---- | ------------------------------------ | --------- | ------------------------------------------------------------------------------------------------- | -------------------------------------------------- | ----------------------------------------------------------------------------------- | -------- |
|      | Authentication & Registration        | Any user  | Register and login securely (email/password and SSO)                                             | Accounts are tied to identities and actions are accountable | Users can sign up/login; tokens/sessions issued; password reset flows implemented    |          |
|      | Role-based Access Control (RBAC)     | System Admin | Assign roles and permissions to users (Citizen, Responder, Authority, Admin)                      | Users only see and act on allowed resources          | Roles exist; endpoints enforce checks; UI hides unauthorized actions                |          |
|      | Location Sharing Consent & Preferences| Citizen/Responder | Opt in/out of live location sharing and set retention preferences                                 | Privacy choices are respected                         | Consent UI; persisted preferences; system enforces opt-outs; audit entries created   |          |
|      | Audit Logging / Change History       | Authority | See an immutable audit trail of incident changes and assignments                                  | Investigations and accountability are possible       | Audit entries for create/update/delete on incidents and assignments; accessible to Admin/Authority |          |
|      | External Partner API (Export/Import) | Research partner | Consume and push incident or aggregated data via a documented API                                  | Partners can integrate and share predictions/resources | OpenAPI spec; sample read/write endpoints with auth; basic tests                   |          |
|      | Offline-capable Mobile Reporting     | Citizen/Responder | Record incident reports offline and sync when online                                               | Reports are not lost due to network outages           | Mobile client queues submissions, retries on connectivity, resolves duplicates     |          |
|      | Push Notifications & Subscription Preferences | Citizen/Responder | Subscribe/unsubscribe and configure notification channels (SMS, push, in-app)                     | Users receive only relevant alerts                    | Preference UI; delivery tests; respects opt-outs; geo-targeting validated         |          |
|      | Shelter Capacity & Reservation Management | Authority/Shelter Manager | Manage shelter capacities, accept reservations and see occupancy levels                           | Citizens can be routed to shelters with availability  | Shelter records with capacity; reservation endpoints and conflict handling        |          |
|      | Resource Reservation & Locking       | Authority/Responder | Reserve a resource for an incident and see its reservation state                                   | Resources are not double-booked                       | Reservation endpoints; locking semantics; expiration policy; UI shows reserved/unavailable |          |
|      | Admin Dashboard & Operational Monitoring | System Admin | Monitor system health, active incident counts, queue lengths and error rates                     | Ops can respond before users are impacted             | Admin UI with health checks, incident metrics and alerts                           |          |
|      | Data Retention & Deletion (GDPR)     | Data Protection Officer | Apply retention policies and delete/anonymize personal data on request                            | Comply with legal privacy requirements                | Retention policy config; delete/anonymize endpoints; logs of deletions            |          |
|      | Search & Filtering for Incidents     | Authority/Responder | Filter and search incidents by status, priority, region, and time range with saved filters        | Quickly focus on relevant incidents                   | Filter UI; performance acceptable for typical dataset; saved filters per user     |          |
